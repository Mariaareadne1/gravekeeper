## 1. WHAT WE ARE BUILDING (the product, the problem, the point)

**Product name:** GraveKeeper (working name). It is a **zombie-agent scanner**: a read-only tool that connects to an organization's cloud and SaaS accounts, inventories every AI agent / automation / non-human identity, and flags the ones that appear **orphaned or dead** — running with live credentials but no active owner and no recent purpose.

**The problem we solve (keep this in mind for every decision):**
- Modern orgs run 25–144 non-human identities (service accounts, API keys, OAuth tokens, AI agents) per human. Median enterprise figure: ~80:1.
- These get created fast and frictionlessly, then abandoned: the employee who built one leaves, the project ends, but the agent keeps running with live access to email, money, and production systems.
- "Zombie agents" / "orphan agents" keep acting on stale logic, become the softest attack target (nobody's watching them), and can do real damage (cf. Knight Capital: a forgotten dormant function fired 4M orders in 45 min, ~$440M lost).
- Nobody can currently answer the basic question: **"How many agents are running in my org, and who owns each one?"** The data exists but is scattered across a dozen consoles that don't talk to each other.

**The point of THIS product:** consolidate those scattered access-control records into one inventory, then infer which agents are orphaned (no recent activity + no living owner + no documented purpose). We are not "tracking the internet." We only ever look inside an environment the user explicitly connects and grants read access to. Discovery is the wedge; it's provable, it changes nothing (read-only), and it requires no trust to try.

**What "dead" means here (be honest about this in code and UI):** we cannot observe death; we INFER it from signals. An identity is a zombie CANDIDATE when one or more hold:
- No activity in audit logs for ≥ N days (default 90).
- Owner no longer exists / is disabled in the identity system (creator left).
- No documented owner at all.
- Over-privileged relative to actual usage (has scopes it never exercises).
We surface candidates with a confidence score and a reason. A human confirms. Never claim certainty.

**Differentiator vs incumbents (Astrix, Entro, Token, Clutch, Oasis, MS Entra Agent ID):** they are enterprise sales-led, birth-governance-focused, and platform-locked. We are **self-serve, free to try, cross-platform, and retroactive** — you can scan your own accounts in minutes and immediately see your zombies. The "zombie" framing + a free public scanner is the wedge nobody else offers.

## 2. TECH STACK (use exactly this)

- **Monorepo** with two apps: `/scanner` (backend) and `/web` (frontend + landing).
- **Backend:** Python 3.11+, FastAPI, async where sensible. Package manager: `pip` with a `requirements.txt` (or `uv` if available). Use `pydantic` v2 for all data models.
- **Frontend:** Next.js 14 (App Router) + TypeScript + Tailwind CSS. This matches the human's existing skill set.
- **Database:** Supabase (Postgres). Use the Supabase Python client in the backend and `@supabase/supabase-js` in the frontend where needed.
- **Cloud connectors (read-only):** `boto3` (AWS), Google API client (GCP + Google Workspace OAuth apps), `msal`/Graph (Azure/Entra) — implement AWS FIRST and most completely; others can be stubbed with a clean interface if time-limited.
- **SaaS connectors (read-only):** GitHub (via `PyGithub` or REST), and a generic OAuth-app-list reader. GitHub FIRST.
- **Testing:** `pytest` for backend, with a synthetic-environment fixture (see Phase 2). `vitest` or Jest + React Testing Library for frontend logic. Playwright optional for e2e if time allows.
- **Lint/format:** `ruff` + `black` (backend), `eslint` + `prettier` (frontend). Run these before every commit.

If any library isn't installed, install it and pin it in the requirements/package files. Keep dependency lists current as you go.

## 2.2 Scoring engine (`scoring.py`) — build and TEST this before connectors
This is pure logic, fully testable offline. Implement `score(agent: AgentRecord, now, thresholds) -> Finding`.
- Rules (each contributes weighted confidence, tune the weights):
  - No activity ≥ 90 days → strong signal.
  - `owner_status` in {missing, disabled} → strong signal.
  - `owner` is None → medium signal.
  - Over-privileged (has broad scopes like admin/`*` but last_activity old or narrow actual use) → medium signal.
  - Combine into a confidence score; set `is_zombie_candidate = confidence >= 0.5` (tunable).
  - Always populate human-readable `reasons`.
- Write `test_scoring.py` with explicit cases: a healthy active agent (not zombie), a clearly dead one (owner gone + no activity → high confidence), a dormant-but-owned edge case, a brand-new agent (no activity yet but recently created → NOT zombie — guard against this false positive explicitly). Ground-truth each expected output.
- Run pytest. All scoring tests must pass.
